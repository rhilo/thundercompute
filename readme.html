FLUX.1-dev LoRA Training Pipeline

Automated pipeline for human-subject LoRA training on FLUX.1-dev. Run on a Thunder Compute GPU instance (e.g. tnr-0): setup environment, clean and crop images, generate captions, synthesize ai-toolkit config, then train.

Pipeline overview

Stage 1setup.sh
Stage 2preprocess.py
Stage 3batch_caption
Stage 4generate_config.py

Place all project files in one directory on the instance (for example ~/flux-lora-pipeline/). After setup, activate the virtual environment before running any Python step.

Prerequisites

  • Ubuntu GPU instance with CUDA (Thunder Compute recommended)
  • python3 (3.10+), git, and network access for pip and Hugging Face
  • Raw training photos packaged as a .zip archive
  • FLUX.1-dev in a diffusers folder (e.g. /home/ubuntu/FLUX.1-dev) or Hugging Face access to black-forest-labs/FLUX.1-dev (see Stage 4)
  • Enough VRAM for JoyCaption inference (caption stage) and ai-toolkit training

Configuration (pipeline.yaml)

Edit one file instead of passing long CLI flags. On first bash setup.sh, pipeline.yaml is created from pipeline.example.yaml (git-ignored so tokens stay local). 

cp pipeline.example.yaml pipeline.yaml   # if needed
nano pipeline.yaml

Quick start

Copy the project to the instance, configure rclone once, then start the TUI. start.sh creates a minimal venv for Textual if setup has not run yet.

rclone config
cd ~/flux-lora-pipeline
bash start.sh

The TUI is the default starting point. Choose Train LoRA for setup, Drive pull, preprocessing, captioning, config generation, training, and LoRA push. Choose Generate Images for ComfyUI model pulls and render pushes.

Advanced CLI runs are still available (each reads pipeline.yaml):

python3 preprocess.py
python3 batch_caption.py
python3 generate_config.py
bash train.sh

Partial runs:

python3 run_pipeline.py --only caption,generate_config,train
python3 run_pipeline.py --from caption

When you switch GPU instances, edit instance: in pipeline.yaml and re-run python3 generate_config.py:

# A100 80GB — full speed
instance:
  profile: a100_80gb
  quantize: false
  gradient_checkpointing: false

# 24GB class cards — VRAM-safe defaults
instance:
  profile: standard_24gb
  quantize: true
  gradient_checkpointing: true

training.model_id must be a diffusers folder or HF repo id, not a lone .safetensors file.

Important: run setup with Bash

Do not use: sh setup.sh
On Ubuntu, sh is dash, which does not support set -o pipefail and will fail immediately.
Use one of:
bash setup.sh  ·  ./setup.sh (after chmod +x setup.sh)
If you accidentally run sh setup.sh, current setup.sh re-executes itself under Bash automatically.

Stage 1 — Environment setup (setup.sh)

Idempotent provisioner: creates .venv, installs PyTorch for CUDA 12.8, pipeline requirements, clones or updates ostris/ai-toolkit into ~/ai-toolkit, then installs ~/ai-toolkit/requirements.txt (includes python-dotenv, diffusers, and other training deps).

bash setup.sh
source .venv/bin/activate

Re-running setup is safe: existing venv and ai-toolkit repo are updated, not duplicated. The ai-toolkit dependency step can take several minutes.

Stage 2 — Dataset preprocessing (preprocess.py)

Extracts images from a zip, crops to 768×1152 portrait (2:3), applies face-aware centering (15% downward shift when a face is detected), rejects blurry frames, and removes near-duplicates. Outputs numbered JPEGs: img_00001.jpg, etc. 

python3 preprocess.py

Settings: paths.raw_zip, paths.dataset_dir, preprocess.blur_threshold, preprocess.hash_threshold in pipeline.yaml.

Stage 3 — Captioning

Default: batch_caption.py (set caption.backend: batch in pipeline.yaml). Uses JoyCaption Alpha Two with batched inference. caption.attn: auto uses SDPA without flash-attn; FA2 is optional via post-setup.py. 

python3 batch_caption.py
# or: python3 run_pipeline.py --only caption

Sequential fallback: caption.backend: sequential → caption.py.

Optional FA2: python3 post-setup.py --max-jobs 3 (or install wheel from Drive venv/wheels/).

Google Drive sync (drive_sync.py)

Sync scope: artificial_generation/image/thunder_compute/ on Drive only. archive/ is never touched by scripts. 

rclone config
bash setup.sh --sync-drive
python3 drive_sync.py pull --profile training --only flux
python3 export_loras.py
python3 drive_sync.py push --profile training --only loras
python3 drive_sync.py promote-loras

Training instance: pull input/, models/FLUX.1-dev/, venv/ → push output/loras/.

ComfyUI instance: bash setup_comfy.sh → pull models/{unet,clip,vae,loras} → push output/images/.

Handoff: trained LoRAs go to output/loras/; curate into models/loras/ for Comfy (manual or promote-loras).

Stage 4 — Training config (generate_config.py)

Writes an ai-toolkit YAML config: LoRA rank/alpha 16, adamw8bit, learning rate 1e-4, bf16 training, float16 saves, quantize: true, portrait buckets [768, 1152]. 

python3 generate_config.py

Model reference (model.name_or_path)

ai-toolkit loads FLUX as a diffusers tree (folders for transformer, vae, text_encoder, etc.). A single ComfyUI flux1-dev.safetensors file cannot be used as name_or_path.

generate_config.py picks, in order:

  1. training.model_id in pipeline.yaml if set
  2. Local diffusers folder: /home/ubuntu/FLUX.1-dev or cached HF snapshot
  3. Otherwise black-forest-labs/FLUX.1-dev (downloads from Hugging Face on first train)

Download the full model locally (recommended if HF hub is slow or gated):

hf auth login
hf download black-forest-labs/FLUX.1-dev --local-dir /home/ubuntu/FLUX.1-dev

Use the hf CLI (not the removed huggingface-cli). Install or upgrade with pip install -U huggingface_hub.

Then regenerate the config so name_or_path points at /home/ubuntu/FLUX.1-dev.

Settings: paths.*, project.trigger_word, training.job_name, training.steps, training.model_id.

Stage 5 — Run training (ai-toolkit)

After config generation, start training with the project wrapper (sets current HF download env vars):

bash train.sh

Reads paths.ai_toolkit_config_out and runtime env from pipeline.yaml. Optional: bash train.sh /path/to/other.yaml --config pipeline.yaml

Checkpoints and samples are written under the training_folder you passed to generate_config.py (e.g. /home/ubuntu/training_run_01). Edit the YAML to change steps, sample prompts, or save frequency before re-running.

Why is network upload high during training?

Your generated config already sets push_to_hub: false and logging.use_wandb: false. Gemini’s guesses about Weights & Biases or automatic Hugging Face checkpoint pushes are not enabled by default in this pipeline.

Typical causes of sustained upload on Thunder/cloud instances:

  1. Hugging Face model fetch (most common early on) — if name_or_path is black-forest-labs/FLUX.1-dev, the hub client (especially with Xet / HF_XET_HIGH_PERFORMANCE) can show large upload counters while syncing chunks, not only download.
  2. Latent caching — cache_latents_to_disk: true writes .safetensors sidecars under your dataset folder. On network-backed home disks, btop may count those writes as outbound traffic.
  3. Checkpoint / sample saves — every 250 steps, LoRA weights and preview images are written under training_folder (local disk I/O, or network storage).

Confirm what your config actually has:

grep -E 'push_to_hub|use_wandb|name_or_path|cache_latents' \
  ~/ai-toolkit/config/flux_lora_v1.yaml

See which remote hosts Python is talking to:

ss -tpn | grep python

After FLUX is fully local, reduce hub traffic:

export HF_HUB_OFFLINE=1
export WANDB_MODE=disabled
bash train.sh ~/ai-toolkit/config/flux_lora_v1.yaml

Use a local diffusers tree (/home/ubuntu/FLUX.1-dev) instead of the HF repo id when generating config.

Troubleshooting

ProblemWhat to do
set: Illegal option -o pipefail Run bash setup.sh, not sh setup.sh
CUDA is required (caption) Run on a GPU instance; confirm nvidia-smi and PyTorch CUDA after setup
flux1-dev.safetensors/transformer or HF connection error Do not set name_or_path to a single .safetensors file. Use black-forest-labs/FLUX.1-dev or download to /home/ubuntu/FLUX.1-dev, then regenerate config
All images rejected in preprocess Lower --blur-threshold or raise --hash-threshold; check zip contents
No module named 'dotenv' (ai-toolkit) Install training deps: pip install -r ~/ai-toolkit/requirements.txt (or re-run bash setup.sh)
Module not found for Python scripts source .venv/bin/activate after setup.sh
Heavy network upload during training Usually not wandb or push_to_hub (both off in generated config). Common causes: Hugging Face Xet traffic while fetching FLUX.1-dev, or latent/checkpoint writes to network-backed disk. Verify with grep -E 'push_to_hub|use_wandb' ~/ai-toolkit/config/flux_lora_v1.yaml and ss -tpn | grep python
HF_HUB_ENABLE_HF_TRANSFER FutureWarning Harmless; training still runs. Re-run bash setup.sh to patch ~/ai-toolkit/run.py, or use bash train.sh
Captions skipped Existing .txt files are cached; set caption.overwrite: true
flash-attn compile OOM python3 post-setup.py --max-jobs 3 or use caption.attn: sdpa (no compile)
rclone / Drive errors Run rclone config; check remote name matches drive.rclone_remote in pipeline.yaml
