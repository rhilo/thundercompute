Thunder Compute FLUX LoRA Pipeline

A complete guide for setting up a GPU instance, copying training files from Google Drive, preparing images, captioning images, creating the training file for ai-toolkit, training a LoRA, sending the LoRA back to Google Drive, and using the LoRA later in ComfyUI.

Read This First

Start the project menu with:
bash tui.sh

The project menu is the screen opened by bash tui.sh. It is also called the TUI, which means terminal user interface. It is not the same thing as the Google Drive setup wizard.

The old-looking screen that asks n/s/q, Storage>, and similar questions is rclone config. That is an outside program used to connect the instance to Google Drive. It is not the project menu.

run_pipeline.py is also not the project menu. It is a command-line runner for people who want to run pipeline steps without using buttons.

If You Are At The Rclone Screen Now

If your terminal currently says Type of storage to configure and shows a long numbered list, choose Google Drive. 

Storage> drive

On older rclone versions, Google Drive may be number 13. You can type either the word drive or the number shown next to Google Drive.

For the earlier prompt that says name>, type this:

name> gdrive

The name gdrive matters because the project config expects that name unless you change it.

The Short Version

Run these commands from a new Thunder Compute Ubuntu instance:

git clone https://github.com/rhilo/thundercompute
cd thundercompute
cp pipeline.example.yaml pipeline.yaml
nano pipeline.yaml
bash tui.sh

Edit pipeline.yaml before training. At minimum, set your trigger word, check the local paths, and decide whether your input images come from Google Drive or a local folder.

What The Project Does

This project has two workflows.

WorkflowUse it whenWhat it does
Train LoRA You have training photos and want to make a FLUX.1-dev LoRA. Pull images and model files, clean images, create captions, create an ai-toolkit training file, train, export the LoRA, and push the LoRA to Drive.
Generate Images You have a ComfyUI instance and want to use trained LoRAs. Pull ComfyUI model folders and LoRAs from Drive, then push rendered images back to Drive.

Important Rules

  • Run commands one at a time. Do not connect project commands with |.
  • Use bash setup.sh, not sh setup.sh.
  • Use bash tui.sh for the project menu.
  • Use python3 run_pipeline.py only when you want the command-line runner.
  • Use rclone config only to connect Google Drive.
  • The training model must be a FLUX diffusers folder or a Hugging Face model name. Do not use one single .safetensors file for training.
  • post-setup.py is optional. You do not need it for normal captioning when caption.attn is auto or sdpa.

In Google Drive

The scripts only work inside this Drive folder:

artificial_generation/image/thunder_compute/

The expected layout is:

artificial_generation/
  image/
    thunder_compute/
      input/              # training images copied down to the training instance
      models/
        FLUX.1-dev/       # full diffusers model folder for training
        unet/             # ComfyUI UNet files

The scripts refuse to touch Drive folders named archive. The working folder is thunder_compute.

One-Time Google Drive Setup

Google Drive access is handled by rclone. This is a separate tool. The project uses it because it can copy large folders between a cloud instance and Google Drive.

Use a current rclone release. The Ubuntu package can be old enough that Google rejects the login page with Access blocked: rclone's request is invalid. The project scripts install current rclone from rclone's official installer instead of relying on Ubuntu's stale package. 

rclone version
curl -fsSL https://rclone.org/install.sh | sudo bash
rclone version

The default remote name is gdrive. Keep these two settings the same:

FileSettingDefault
pipeline.yamldrive.rclone_remotegdrive
drive.manifest.yamlrclone_remotegdrive

Start the wizard

rclone config

Exact answers

n                         # New remote
name> gdrive              # Must match pipeline.yaml and drive.manifest.yaml
Storage> drive            # Or type the number shown next to Google Drive
client_id>                # Press Enter
client_secret>            # Press Enter
scope> 1                  # Full Drive access, needed for copy down and copy up
root_folder_id>           # Press Enter
service_account_file>     # Press Enter
Edit advanced config? n
Use auto config? n        # Use n on a rented cloud instance

# rclone prints a Google login URL.
# Open that URL in your normal browser, approve access, then paste the code back into SSH.

Configure this as a Shared Drive? n
Keep this remote? y
q                         # Quit config

Check that it works

rclone lsd gdrive:
python3 drive_sync.py check

If the first command cannot list your Drive, the project cannot copy files yet. Re-run rclone config and check that the remote is named gdrive.

If Google shows Access blocked, stop the rclone wizard with Ctrl+C, install current rclone with the command above, then run rclone config again.

Script Reference

tui.sh

tui.sh is the easiest entry point. It opens the project menu.

bash tui.sh

It calls start.sh.

start.sh

start.sh prepares enough Python packages to show the project menu.

It does these things:

  1. Installs rclone with apt if it is missing.
  2. Warns if the Google Drive remote is not configured.
  3. Creates .venv if it does not exist.
  4. Installs textual and PyYAML if the menu needs them.
  5. Starts pipeline_tui.py.

It does not install PyTorch, clone ai-toolkit, or start training.

setup.sh

setup.sh prepares the machine for training.

bash setup.sh
bash setup.sh --sync-drive
bash setup.sh --no-sync

It does these things:

  1. Creates pipeline.yaml from pipeline.example.yaml if missing.
  2. Creates .venv if missing.
  3. Installs PyTorch for CUDA 12.8.
  4. Installs this repo's Python packages.
  5. Clones or updates ~/ai-toolkit to a fixed commit.
  6. Installs ai-toolkit Python packages.
  7. Patches one ai-toolkit environment variable warning.
  8. If --sync-drive is used, checks Google Drive and pulls training input, FLUX files, and optional wheel files.

Run it again when packages are broken or when a new instance has no training environment yet.

drive_sync.py

drive_sync.py copies files between Google Drive and the instance.

python3 drive_sync.py check
python3 drive_sync.py status --profile training
python3 drive_sync.py pull --profile training --only input,flux,venv
python3 drive_sync.py push --profile training --only loras
python3 drive_sync.py promote-loras
python3 drive_sync.py pull --profile comfyui
python3 drive_sync.py push --profile comfyui --only images

It reads drive.manifest.yaml. It does not read every Drive path from pipeline.yaml.

preprocess.py

preprocess.py turns original images into training-ready images.

python3 preprocess.py

It does these things:

  1. Looks for images in paths.input_dir.
  2. If that folder is not available, it can use paths.raw_zip.
  3. Finds common image types such as jpg, png, webp, bmp, tif, and tiff.
  4. Crops images to portrait shape, 768 by 1152.
  5. Tries to keep faces placed well in the crop.
  6. Rejects images that are too blurry.
  7. Rejects near-duplicate images.
  8. Writes files named like img_00001.jpg into paths.dataset_dir.

Change blur and duplicate behavior in pipeline.yaml, not with command-line flags.

batch_caption.py

batch_caption.py creates captions for many images at a time on the GPU.

python3 batch_caption.py
python3 batch_caption.py --attn sdpa
python3 batch_caption.py --batch_size 32

It does these things:

  1. Reads cleaned images from paths.dataset_dir.
  2. Skips images that already have a matching .txt file unless overwrite is enabled.
  3. Loads JoyCaption Alpha Two.
  4. Creates captions in batches.
  5. Writes one .txt caption beside each image.
  6. Adds your project.trigger_word to the start of each caption.

caption.py

caption.py is the slower caption fallback. It captions one image at a time.

Use it by setting this in pipeline.yaml:

caption:
  backend: sequential

post-setup.py

post-setup.py is optional. It tries to install Flash Attention for faster captioning.

python3 post-setup.py --max-jobs 3
python3 post-setup.py --low-memory
python3 post-setup.py --skip-flash-attn

If it takes too long or fails, set this and keep going:

caption:
  attn: sdpa

generate_config.py

generate_config.py writes the training file used by ai-toolkit.

python3 generate_config.py

It checks the cleaned dataset folder, chooses the model path, writes LoRA settings, and saves the file at paths.ai_toolkit_config_out.

train.sh

train.sh starts ai-toolkit training.

bash train.sh
bash train.sh /path/to/ai-toolkit-config.yaml --config pipeline.yaml

It activates the project environment, applies settings from pipeline.yaml, enters ~/ai-toolkit, and runs ai-toolkit.

export_loras.py

export_loras.py gathers trained LoRA files so they can be uploaded.

python3 export_loras.py
python3 export_loras.py --export-dir /home/ubuntu/export/loras

It searches paths.training_output_dir for .safetensors files and copies them to drive.export_loras_dir.

setup_comfy.sh

setup_comfy.sh is for a ComfyUI instance, not a training instance.

bash setup_comfy.sh
bash setup_comfy.sh --sync-only

It installs rclone if needed, checks Drive, pulls ComfyUI model folders, and warns if expected folders are empty.

run_pipeline.py

run_pipeline.py runs training stages without the menu.

python3 run_pipeline.py
python3 run_pipeline.py --from caption
python3 run_pipeline.py --only caption,generate_config,train
python3 run_pipeline.py --best
python3 run_pipeline.py --hardware-preset a100_80gb_8c64g

It runs these stages in order: preprocess, caption, generate config, train.

It does not export LoRAs and does not push anything to Drive.

What Happens Inside Each Training Stage

1. Setup

The machine gets Python packages, PyTorch, and ai-toolkit. No captions are created here. No training happens here.

2. Pull Files From Drive

Training input images and model folders are copied from Google Drive to the instance. Existing folders are normally skipped so the same copy does not happen again.

3. Preprocess Images

Original images are opened, cropped, filtered, and saved as clean training images.

4. Caption Images

Each clean image gets a text file beside it. The text file is what the trainer reads along with the image.

5. Create Training File

The repo creates an ai-toolkit YAML file. This file tells ai-toolkit where the dataset is, where to save output, what model to use, and what training settings to use.

6. Train

ai-toolkit reads the generated YAML file and trains the LoRA. Output is written under paths.training_output_dir.

7. Export

The repo copies LoRA files from the training output folder into a smaller export folder.

8. Push To Drive

The export folder is copied to Google Drive under output/loras.

9. Promote For ComfyUI

LoRAs in Drive output/loras are copied to Drive models/loras. ComfyUI instances pull from models/loras.

Model Files

Training needs FLUX.1-dev as a diffusers folder. A diffusers folder contains subfolders such as transformer, vae, and text encoder folders.

A single file such as flux1-dev.safetensors is useful for ComfyUI, but it is not the correct training input for ai-toolkit in this repo.

To download the full model with the Hugging Face CLI:

hf auth login
hf download black-forest-labs/FLUX.1-dev --local-dir /home/ubuntu/FLUX.1-dev

Then set this in pipeline.yaml:

training:
  model_id: /home/ubuntu/FLUX.1-dev

ComfyUI Workflow

This repo does not install ComfyUI. It only copies model files and rendered images for a ComfyUI instance.

On a ComfyUI instance:

cd ~/thundercompute
bash setup_comfy.sh

This pulls:

  • Drive models/unet to /home/ubuntu/ComfyUI/models/unet
  • Drive models/clip to /home/ubuntu/ComfyUI/models/clip
  • Drive models/vae to /home/ubuntu/ComfyUI/models/vae
  • Drive models/loras to /home/ubuntu/ComfyUI/models/loras

After generating images in ComfyUI, upload them with:

python3 drive_sync.py push --profile comfyui --only images
